# AI Tools Platform — план реализации

Дата: 2026-07-19
Статус: черновик v1 (для обсуждения перед стартом разработки)

## 1. Видение

Единая платформа на Vercel с набором AI-инструментов, закрывающих весь цикл разработки фичи:

- бизнес-требования
- системный анализ / проектирование
- дизайн
- код (в идеале без участия разработчика — разработчик занимается только проектированием и системным анализом)
- код-ревью (уже есть отдельная тулза, будет интегрирована и доработана)
- автотесты (качество и покрытие)

Каждый инструмент работает самостоятельно, но резко выигрывает в качестве, если у платформы есть контекст проекта: загруженная документация, код, история фич. Пользователь один раз настраивает workspace (грузит документы, прописывает интеграции, выбирает модели) — дальше платформа его "помнит": история прогонов, незавершённые фичи, промпты, настройки.

## 2. Стек (зафиксировано)

- **Next.js (App Router) + TypeScript** — единое full-stack приложение, деплой на Vercel как один проект.
- **Vercel Postgres (Neon) + расширение `pgvector`** — основная БД и хранилище эмбеддингов в одном месте. ORM — Drizzle (лучше дружит с pgvector, чем Prisma).
- **Vercel Blob** — хранилище сырых загруженных файлов (документы, экспорт confluence, код и т.п.).
- **Vercel AI SDK** — унифицированный слой поверх LLM-провайдеров: по умолчанию Anthropic, но с поддержкой произвольного `baseURL` + токена, которые прописывает пользователь.
- **Оркестрация многошаговых/фоновых задач** (инжест документов, полный цикл фичи по стадиям) — отдельный слой очереди/воркфлоу (Inngest или Trigger.dev), т.к. serverless-функции Vercel плохо подходят для долгих синхронных процессов (парсинг больших пачек документов, эмбеддинги, многошаговые цепочки инструментов).
- **Авторизация** — на старте однопользовательский режим (без экрана логина), но модель данных сразу проектируется с `workspaceId`/`userId`, чтобы потом безболезненно добавить Clerk/Auth.js и многопользовательность.

## 3. Важный принцип: секреты не хранятся

Пользователь вводит:

- URL и токен GitLab
- URL и токен LLM-провайдера (если не дефолтный Anthropic)

Эти токены **никогда не пишутся в БД**. Они держатся только в зашифрованной httpOnly-сессионной cookie (например, через `iron-session`) на время браузерной сессии и передаются на сервер только при вызове конкретного инструмента. Если сессия истекла — токен нужно ввести заново. Всё остальное (URL без токена, выбор модели, дефолтные промпты, загруженные документы) сохраняется постоянно.

## 4. Модель данных (черновая)

- `workspace` — единица "рабочего пространства" пользователя (в MVP фактически одна запись)
- `document` — загруженный файл: имя, ссылка на Blob, статус обработки (processing/ready/error), тип
- `document_chunk` — чанк текста документа + `embedding vector` + метаданные (источник, раздел)
- `tool_settings` — на каждый инструмент: выбранная модель, кастомный provider URL (без токена)
- `prompt_template` — привязан к инструменту: название, текст, признак "дефолтный/кастомный", признак активности
- `run` — единичный запуск инструмента: какой инструмент, какая модель, промпт, вход/выход, `input_tokens`/`output_tokens`, оценка стоимости, статус, timestamp
- `feature_workflow` — сущность "фича", которую ведём по полному циклу: текущая стадия, статус, ссылки на связанные `run`
- (позже) `user` — для многопользовательского режима, пока не обязателен

## 5. Общий контракт AI-инструмента

Чтобы инструменты были и независимыми, и частью общего пайплайна, у каждого — единый интерфейс:

- `key`, `name`, `description`
- `defaultPrompts()` — набор дефолтных промптов из коробки
- `run(input, { promptId, model, useProjectContext })` → результат + `tokenUsage`
- `estimateCost(model, avgTokens)` — для вкладки статистики
- флаг `benefitsFromContext` — использует ли граф/векторный поиск по документам

Такой контракт позволяет добавлять новые инструменты как модули, не трогая ядро платформы.

## 6. Контекст проекта и "граф зависимостей"

Реализация на MVP-этапе: классический RAG поверх pgvector, а не отдельная graph DB (проще внедрить и поддерживать, дешевле по инфраструктуре).

**Форматы на старте: только `.md`.** Остальное (pdf, docx, confluence export, репозиторий кода) — сознательно отложено, вернёмся к этому отдельным заходом, когда каркас заработает на md.

Пайплайн ingestion для `.md`:

1. **Извлечение** — `.md` уже текстовый формат, парсер не нужен в смысле "распознать бинарник" (как для pdf/docx), но нужно разобрать структуру:
   - вынуть YAML frontmatter, если есть (`---\ntitle: ...\n---`) — это готовые метаданные документа;
   - построить дерево заголовков (`#`, `##`, `###`) — оно нужно, чтобы знать, из какого раздела взят кусок текста;
   - код-блоки (```) и таблицы не разбивать посередине — если чанк режется, граница должна проходить по границе блока, иначе эмбеддинг куска кода без контекста бесполезен.
2. **Чанкование** — не резать текст "вслепую" по N символов, а резать по секциям (по заголовкам), и уже внутри длинной секции — по абзацам с ограничением по размеру (например, ~500–800 токенов) и небольшим перекрытием (overlap), чтобы не терять контекст на границе чанка.
3. **Метаданные на чанк** — путь к исходному файлу, "хлебные крошки" заголовков (`H1 > H2 > H3`), номер чанка внутри документа. Это нужно, чтобы потом в ответе инструмента можно было сослаться на источник ("см. раздел X документа Y").
4. **Эмбеддинг** — каждый чанк прогоняется через embeddings-эндпоинт LLM-провайдера, вектор + текст + метаданные пишутся в `document_chunk` (pgvector).
5. При генерации нового документа (например, ТЗ на фичу) — retrieval релевантных чанков + явные связи "фича X ссылается на документы Y, Z" сохраняются в `feature_workflow`, это и есть практический "граф зависимостей" на старте.
6. Полноценную graph DB (Neo4j) можно рассмотреть позже, если retrieval-подхода перестанет хватать для сложных явных связей между сущностями.

## 7. Каркас платформы (MVP) — с этого начинаем

Порядок: сначала общая платформа, потом в неё встраивается код-ревью тулза как первый реальный инструмент.

1. **Layout и навигация** — список инструментов, переход между ними.
2. **Настройки** — LLM provider URL + токен (сессионно), GitLab URL + токен (сессионно), выбор модели на инструмент (дефолт — Anthropic).
3. **Загрузка документов** — upload → Blob → фоновый инжест (чанки + эмбеддинги) → статус готовности.
4. **История** — список прошлых прогонов и фич, включая незавершённые, с возможностью вернуться и продолжить.
5. **Вкладка "Промпты"** (на каждый инструмент) — дефолтные промпты + создание/редактирование своих, выбор активного.
6. **Вкладка "Статистика"** — средний расход токенов и оценка стоимости на запрос по каждому инструменту и модели.
7. **Универсальный "раннер" инструмента** — экран запуска: выбор промпта/модели, опция "использовать контекст проекта", вывод результата, сохранение в историю.

## 8. Дальнейшие фазы (после каркаса)

- **Фаза 2** — интеграция существующей код-ревью тулзы как первого модуля в общую платформу (нужно посмотреть код/API текущей реализации).
- **Фаза 3** — улучшение код-ревью по метрикам (когда будут данные о качестве).
- **Фаза 4** — по очереди, отдельными циклами проектирования: автотесты (качество/покрытие), автокодинг, системный анализ/проектирование, бизнес-требования, дизайн. Каждый — отдельное глубокое погружение, как договорились.
- **Фаза 5** — сквозная оркестрация: полный цикл ведения фичи, где стадии передают контекст друг другу через `feature_workflow`.

## 9. Осознанно не делаем на старте (non-goals MVP)

- Многопользовательская авторизация и роли
- Биллинг/платежи
- Командный доступ / шаринг workspace
- Полноценная graph DB

Схема данных не должна блокировать добавление этого позже.

## 10. Ближайшие шаги

1. Инициализировать Next.js + TypeScript проект, подключить Vercel Postgres (pgvector) и Blob.
2. Спроектировать и накатить схему БД (см. раздел 4).
3. Собрать настройки + загрузку документов + фоновый инжест.
4. Собрать вкладки "Промпты" и "Статистика" + универсальный раннер инструмента.
5. Как только будет прислан код существующей код-ревью тулзы — спроектировать её интеграцию как первого модуля.

## 11. Ориентировочная стоимость инфраструктуры (однопользовательский режим, июль 2026)

Оценка на старте, при лёгкой нагрузке одного пользователя. Актуальные цены проверялись на официальных страницах провайдеров.

| Статья расходов | Тариф | Ориентировочно в месяц |
|---|---|---|
| Vercel Pro (подписка) | $20/мес, в подписку уже включён $20 usage-кредит на compute/bandwidth/Blob | $20 (в лёгком режиме overage почти нет — кредит его покрывает) |
| Vercel Blob (файлы) | $0.023/GB-мес хранение + отдельная плата за трафик, сверх бесплатных лимитов — оплачивается из кредита Pro | ~$0 сверху при небольшом объёме md-файлов |
| Postgres + pgvector (Neon, через Vercel Marketplace) | Free-тариф: 0.5 GB хранилища + 100 CU-часов/проект; дальше pay-as-you-go — $0.106/CU-час + $0.35/GB-мес, без минимального платежа | $0 на старте, закладывать $5–10 на вырост |
| Фоновые задачи (Inngest) | Free (Hobby): 50 000 выполнений/мес, 5 параллельных — с запасом для одного пользователя; Pro — $99/мес, но он не нужен на этом масштабе | $0 |
| LLM-вызовы (Anthropic API) | Claude Sonnet: $2 / $10 за 1M токенов вход/выход (акционная цена до 31.08.2026, потом $3/$15); Haiku: $1/$5; Opus: $5/$25 | сильно зависит от объёма использования — см. пример ниже |
| Эмбеддинги документов | У Claude нет своего embeddings API, Anthropic рекомендует Voyage AI: voyage-3-lite $0.06 / voyage-3-large $0.18 за 1M токенов | центы даже на десятках md-документов |

**Пример расчёта LLM-затрат:** прогон код-ревью тулзы с контекстом ~20K входных токенов и ~2K выходных на Sonnet ≈ (20 000/1 000 000)×$2 + (2 000/1 000 000)×$10 ≈ **$0.06 за прогон**. При 100–300 прогонах в месяц по всем инструментам это ≈ $6–20/мес, при более активном использовании — соответственно больше. Это самая переменная и непредсказуемая статья расходов, поэтому вкладка "Статистика" в каркасе платформы (раздел 7, п.6) как раз и нужна, чтобы этот параметр было видно по факту, а не только в теории.

**Итог по инфраструктуре без учёта LLM-вызовов:** реалистично $20/мес (по сути только подписка Vercel Pro — Neon и Inngest на старте укладываются в free-тарифы). **С учётом LLM-вызовов** при умеренном личном использовании — ориентировочно **$30–60/мес**, дальше зависит от того, сколько прогонов инструментов в месяц и какие модели выбраны (Sonnet дороже Haiku примерно в 2 раза).
